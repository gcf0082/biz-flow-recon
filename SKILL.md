# biz-flow-recon

把代码讲给安全测试人员听——按子功能 + 接口的粒度，用最清晰的形式说"它在做什么"：文件 I/O、外部命令、第三方交互、加解密。**只描述事实**，判断权留给读者。

## 何时使用 / 何时不用

**使用**——只要用户表达出"想搞清这个代码在做什么"的意图，无论是否提到"安全"：
- "梳理业务流"、"讲讲这个项目 / 模块 / 接口"、"给我个 brief"、"帮我了解一下"
- "渗透测试前摸底"、"代码审计前盘点"、"接口/功能盘点"
- 具体到类或接口："讲讲 XxxController 在干什么"、"POST /api/yyy 这个接口怎么走的"
- 主要面向 **Java 后端**，前端与全栈也支持

**不使用**——这些走 `/security-review`：在做漏洞分析、写修复建议、写报告，或问"有没有 SQL 注入 / 越权 / XSS"等具体漏洞。

## 原则

- **描述事实，不评判**。禁用"风险/漏洞/不安全/缺乏校验/可能被攻击/建议加固"等词。
- **清晰优先，不拘形式**。按内容挑：图 / 编号 / labeled 短列表 / 句子 / 一行话。**禁止机械填表**（每条都 `- 入口:` `- 触发者:`），**禁止空字段**（"外部命令: 无"）。
- **优先跨信任边界的动作**（用户输入入口、I/O、外呼、加解密），而非内部纯计算。
- **能下钻就下钻**：调脚本读脚本、调内部接口找接收方、引用 SQL 读 SQL。
- **找不到必须报**：在描述里点名，并在产物末尾的 `## 未跟到的引用` 节集中列出。**不编、不漏、不静默吞掉**。
- **大就拆**：一次装不下时按子功能拆子任务、逐个产出、最后汇总。
- **复杂时配图**：多服务/多分支/异步/状态流转时，在描述后追加一个 mermaid 图。
- **捕请求**：REST/RPC/MQ 等结构化输入必须交代请求维度——path 变量 / query / 关键 header（鉴权、Content-Type）/ body。body 是 JSON 时给字段树（记号见步骤 4）；DTO 类型要下钻到定义读出真实字段，不要猜。

## 取舍

**保留**：用户输入入口、文件读写、外部进程、网络外呼、加解密/签名、鉴权与会话、数据持久化。
**砍掉**：CSS/i18n/纯计算/单测/构建脚本/健康检查/静态资源/Spring Actuator 等纯运维端点。

## 方法

### 0. 加载工作目录

在 `<cwd>/biz-flow-recon/` 找：若有 `knowledge/` 就**读完所有 .md** 当先验；模板路径优先走 `<cwd>/biz-flow-recon/templates/{name}.md`，回退到 skill 包内默认。`output/` 等到要写产物时再建。

### 1. 定粒度

- **粒度 B（默认）**：子功能 + 接口，模板 `default.md`。
- **粒度 A**：用户明确说"整项目/全量梳理/系统全貌"时，模板 `overview.md`。
- **粒度 C**：用户给了具体 URL 或 类#方法 时，模板 `endpoint.md`。

### 2. 盘入口 + 判规模

找 `@RestController`/`@Controller`/`@RequestMapping`、`@Scheduled`、MQ 监听器、CLI `main`、前端 router。盘完后判规模：能一次装下就继续步骤 3；装不下（经验参考：6+ 子功能 / 25+ 接口 / 多模块）就进**大项目流程**（步骤 5 末尾详述）。

### 3. 跟链路 + 下钻

每个入口跟到底（Service/Mapper/配置/I/O/进程/网络/加密点）。**遇到指向更深层代码的引用就跟**——常见目标：进程脚本、SQL 文件、内部 HTTP/RPC、消息 topic、反射类、动态加载的规则文件。

- **边界**：当前仓 / 本地兄弟目录 → 必跟；跨仓 → 仅 `knowledge/conventions.md` 指明位置时跟；真正外部（第三方/SaaS）→ 描述对外协议即可。
- **深度**：默认 2 层；第 3 层及以后一句话概括。`conventions.md` 写明深度时按用户。
- **走不通**：在描述里点名"未在工作区找到 X"，并加入末尾"未跟到的引用"节。

### 4. 写描述

每条接口必须交代：哪段代码的哪个入口（含 `类#方法，文件:行号`）、谁触发、**请求参数**（REST/RPC/MQ 等结构化输入；记号见下）、当**实际发生**时讲清文件 I/O / 命令 / 第三方 / 加解密——没发生的不写。形式按流程特点挑：

- **mermaid 图**：多参与方 / 分支 / 异步——`sequenceDiagram` 用于多方调用链（最常用），`flowchart TD` 用于分支，`stateDiagram-v2` 用于状态机。一图最多一张，主干 + 关键分支。
- **编号步骤**：内在顺序操作。
- **labeled 短列表**：多类独立事实，写成 `**第三方**: ...` `**加解密**: ...`，**只列发生过的**。
- **几句话**：自然能串成句子时。
- **一行话**：简单读接口。

下钻得到的下游事实写在同一条描述里。**不要嵌套子标题**。

**请求参数记号**（Body 给字段树时用，紧凑统一）：

- `string!` / `int!` —— `!` 表示非空/必填（来自 `@NotNull`/`@NotBlank` 等）
- `string?` —— 可选/可为空
- `[ ... ]` —— 数组
- `// ...` —— 行内注释，写约束（长度、正则、范围）或字段含义
- 末尾挂 DTO 全限定名 + `文件:行号`

简单 query 1 行 inline（`**请求**: query \`?page=1&size=20\` ...`）；嵌套或字段多用 fenced 代码块（` ```json5 ... ``` `）。GET 没 body 就不写 body 行。

### 5. 写产物

落到 `<cwd>/biz-flow-recon/output/`，按粒度命名：

- 粒度 A 单次：`overview.md`
- 粒度 B 全量：`features.md`；限定范围：`features-{slug}.md`
- 粒度 C：`endpoint-{METHOD}-{path-slug}.md`
- 大项目：`_plan.md` + 多份 `features-{slug}.md` + `overview.md`

slug 小写连字符，中文转拼音/英文同义词。**默认覆盖同名文件**，便于 git diff。写完告诉用户路径。

**大项目流程**（步骤 2 判出"装不下"时走）：

a. 按目录/包/模块划分子功能，写 `output/_plan.md`（格式见模板），简短告诉用户拆分方案。
b. 逐个子任务跑步骤 3-4，scope 限定到该子功能，产出 `output/features-{slug}.md`。默认串行；`conventions.md` 写"执行模式: 并行"时用 Task 工具并行派发 subagent。
c. 全部完成后写 `output/overview.md`：核心总结 + 子功能索引（链接到各 features-X.md）+ 未跟到的引用汇总。
d. **中断恢复**：重跑默认跳过已存在的 `features-*.md`；要重做先删对应文件。

## 工作目录与模板

`docs/workspace-layout.md` 详述被分析项目里 `biz-flow-recon/{knowledge,templates,output}/` 的约定。**每次输出前先读对应模板再照样写**，不要凭记忆——这样用户改了模板能立即生效。
