# biz-flow-recon

像一位熟悉系统的工程师，把代码讲给安全测试人员听。先一段连贯的话讲清"它在做什么"，再按子功能分组，每个接口下用**自然的段落**讲它怎么跑——文件 I/O、外部命令、第三方交互、加解密自然地织进句子里。判断权留给读者。

## 原则

- **描述事实，不评判**。禁用"风险/漏洞/不安全/缺乏校验/可能被攻击/建议加固"等词。
- **每条业务流用 1-2 段自然语言讲，不用 bullet**。事实嵌在句子里，简单接口可一句话讲完。
- **不存在的事实不提**，让产物像讲解、不像填表。
- **优先跨信任边界的动作**（用户输入入口、I/O、外呼、加解密），而不是内部纯计算。
- **能下钻就下钻**。看到调脚本就读那个脚本、调内部接口就找接收方代码、引用 SQL 文件就读 SQL，把下游实际做了什么编织进**同一段**描述。
- **找不到必须报**。任何下钻目标（脚本、内部服务、SQL、反射类……）在工作区找不到时，**必须在出现该引用的描述里直接点名**，并在产物末尾的 `## 未跟到的引用` 一节集中列出（含调用点 `文件:行号` 和找不到的目标）。**不编、不漏、不静默吞掉**——这是安全测试的盲点情报。
- **大就拆**。一次分析装不下整个范围时，自动按子功能拆子任务、逐个产出，最后汇总。不要硬塞进一次分析里、漏接口、爆上下文。

## 取舍

**保留**：用户输入入口、文件读写、外部进程调用、网络外呼、加解密/签名、鉴权与会话、数据持久化。

**砍掉**：纯 CSS/样式、i18n 文案、内部纯计算函数、单元测试、构建脚本、健康检查、静态资源、Spring Actuator 之类纯运维端点。

## 方法

### 步骤 0：加载工作目录（每次都做）

1. 在当前工作目录下找 `biz-flow-recon/`：
   - 如果存在 `biz-flow-recon/knowledge/`，**把里面所有 `.md` 都读一遍**当先验知识，分析时尊重其中的术语、约定、模块说明。
   - 模板路径按这个顺序解析：`<cwd>/biz-flow-recon/templates/{name}.md`（项目覆盖）→ `<skill-dir>/templates/{name}.md`（默认）。
2. 如果当前目录下还没有 `biz-flow-recon/output/`，分析完后再创建——别预先建。

### 步骤 1：定粒度（默认 B）

- **默认走粒度 B**：按子功能分组（登录、下单、文件上传……），每组下列出相关接口。模板用 `templates/default.md`。
- **粒度 A（整项目）**：仅当用户明确说"整个项目""全量梳理""系统全貌"等总览意图时使用。模板 `templates/overview.md`。
- **粒度 C（单接口）**：仅当用户给了具体 URL 或类#方法名时使用。模板 `templates/endpoint.md`。

### 步骤 2-4：分析

2. **盘入口**：在选定范围内找 `@RestController`/`@Controller`/`@RequestMapping`、`@Scheduled`、MQ 监听器、CLI `main`、前端 router。
3. **跟链路并下钻**（关键）：每个入口跟到底——Service、Mapper/Repository、配置文件（`application.yml`）、文件 I/O、外部进程、网络调用、加密点。

   每遇到一个**指向更深层代码**的引用，尝试在工作区找到目标读它，把它做了什么也作为同一段叙述的一部分写出。常见下钻点（让 Opus 自己识别，这里只是提醒）：
   - 进程调用：`Runtime.exec` / `ProcessBuilder` 引用的脚本和命令
   - SQL：MyBatis Mapper XML、`@Query`、`@Procedure`、本地 `.sql` 文件
   - 内部 HTTP/RPC：URL 形如 `http://<内部域名>/...`、Feign 客户端、gRPC stub
   - 消息：Producer 发往的 topic → 找同仓的 Consumer
   - 反射/SPI/插件：`Class.forName`、`ServiceLoader`、Spring `@Conditional`
   - 二级配置：动态加载的脚本/规则文件

   **边界**：
   - 当前仓 / 本地兄弟目录可读 → 必须跟
   - 跨仓 → 仅当 `knowledge/conventions.md` 指明位置时跟（如 "charge-svc → ../services/charge-svc/"）
   - 真正外部（第三方/SaaS） → 描述对外协议即可，按黑盒处理

   **深度**：默认下钻 **2 层**（业务接口 → 直接被调的下游 → 它再调的下一层）。第 3 层及以后用一句话概括"再往下还会 X、Y"，不展开。`knowledge/conventions.md` 写明深度时按用户的。

   **走不通**：在出现该引用的描述里**直接点名**"未在工作区找到 X"（不要装作没看见，不要编造），并把这条加入产物末尾的 `## 未跟到的引用` 节。

4. **写段落**：照模板的骨架，把抓到的事实**编织成一两段散文**讲清楚，自然提到 `(类#方法，文件:行号)` 作为可定位锚点。下钻得到的下游事实**继续织进同一段**，用从句串接，不要起新标题、不要嵌套列表。

### 步骤 2.5：判规模 + 拆子任务（关键）

步骤 2 盘出入口后，**先判规模**：

- 一次分析就能讲清所有子功能、不漏细节、不爆上下文 → 走小项目流程（继续步骤 3-4-5，输出一份 `features.md`）
- 否则 → 进入**大项目流程**

经验参考：6+ 个子功能 / 25+ 个接口 / 多 maven 模块 / 用户明确说"整项目梳理"——多半要拆。具体让自己拿捏，**不要硬塞**。

**大项目流程**：

a. **盘子功能**，按目录/包/maven 模块划分，写 `output/_plan.md` 列出子任务。格式见 `templates/_plan.md`：每条含 `name`、`slug`、`scope`（一组目录路径）、`output`（目标文件名）。

b. **简短告诉用户**拆分方案，例如："项目较大，我打算拆成 6 个子任务：认证、订单、支付……，每份产出 `features-{slug}.md`，最后汇总 `overview.md`。"

c. **逐个执行子任务**：对每个子任务跑步骤 3-4，scope 限定在该子功能的代码范围内，产出 `output/features-{slug}.md`（含"未跟到的引用"节）。
   - **默认串行**——稳。
   - 子任务彼此独立时，可在 `knowledge/conventions.md` 写 `执行模式: 并行` 打开；这时用 **Task 工具并行派发 subagent**，每个 subagent 拿到 skill 路径 + scope + 目标文件名，独立产出自己那份。

d. **写 overview.md** 汇总：核心功能总结（150-300 字）+ 子功能索引（每条链接到对应 `features-X.md`）+ 未跟到的引用汇总（合并各 features 末尾那一节）。

e. **中断恢复**：重跑时**默认跳过已存在的 `features-*.md`**，只做没做完的。要重做某条，让用户先删对应文件。用户说"全部重做"才覆盖。

### 步骤 5：写产物

按下面的文件名约定写到 `<cwd>/biz-flow-recon/output/`（目录不存在就建）：

| 粒度 | 文件名 |
|---|---|
| A 整项目（小项目，单次分析） | `overview.md` |
| A 整项目（大项目，自动拆分） | `_plan.md` + 多份 `features-{slug}.md` + `overview.md` |
| B 子功能（默认全量） | `features.md` |
| B 子功能（限定范围，如"支付下单流程"） | `features-{slug}.md`（如 `features-pay-order.md`） |
| C 单接口 | `endpoint-{METHOD}-{path-slug}.md`（如 `endpoint-POST-api-login.md`） |

slug 规则：小写、连字符；中文/特殊字符转拼音或英文同义词。**默认覆盖同名文件**，便于 git diff。

写完告诉用户产物路径。

## 段落要包含什么

每条接口的描述段必须自然交代：

- 是哪段代码的哪个入口（含 `文件:行号`）
- 谁在触发
- 当**实际发生**时把以下信息织进描述：读写了哪些文件（路径+格式）、跑了哪些命令（含目的）、调了哪些第三方（协议、字段、鉴权）、做了哪些加解密（算法+对象+为什么）

不存在的不写。读者读完一段，应能跳到代码并知道这段流程在做什么。

## 工作目录约定

详见 `docs/workspace-layout.md`。简言之：被分析项目根目录下放 `biz-flow-recon/{knowledge,templates,output}/`，前两个用户填，最后一个 skill 写。

## 输出格式

格式由 `templates/` 下的模板定义，用户可在项目里覆盖。每次输出前**先读相应模板**再照样写，**不要凭记忆写**——这样用户改了模板能立即生效。
