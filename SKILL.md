# biz-flow-recon

面向安全测试人员说明代码行为——按子功能 + 接口的粒度，以最清晰的形式呈现"它在做什么"：文件 I/O、外部命令、第三方交互、加解密。**仅描述事实**，安全风险判断由读者自行做出。

## 何时使用 / 何时不用

**使用**——用户表达出"需要了解这段代码功能"的意图即可触发，无论是否提及"安全"：
- "梳理业务流"、"讲讲这个项目 / 模块 / 接口"、"给我个 brief"、"帮我了解一下"
- "渗透测试前摸底"、"代码审计前盘点"、"接口/功能盘点"
- 具体到类或接口："讲讲 XxxController 在干什么"、"POST /api/yyy 这个接口怎么走的"
- 主要面向 **Java 后端**，前端与全栈亦支持

**不使用**——以下场景应使用 `/security-review`：进行漏洞分析、撰写修复建议、产出安全报告，或针对"是否存在 SQL 注入 / 越权 / XSS"等具体漏洞的提问。

## 原则

- **描述事实，不做评判**。禁用"风险/漏洞/不安全/缺乏校验/可能被攻击/建议加固"等词。
- **清晰优先，不拘形式**。按内容选择最清晰的呈现：示意图 / 编号步骤 / 字段化短列表 / 连贯叙述 / 一句话概述。**禁止机械化字段堆砌**（每条都 `- 入口:` `- 触发者:` 的固定模板抄写），**禁止空字段**（"外部命令: 无"）。
- **优先跨信任边界的动作**（用户输入入口、I/O、外呼、加解密），而非内部纯计算。
- **尽可能递归追溯**：调用脚本时分析脚本内容、调用内部接口时定位接收端代码、引用 SQL 文件时读取 SQL。
- **未能定位必须显式标注**：在描述中点名，并在产物末尾的 `## 未能追溯的引用` 节集中列出。**不得编造、不得遗漏、不得静默忽略**。
- **每接口一个 subagent**（强制规则）：**每个接口的分析必须由独立的 Task subagent 完成**——彼此独立、可并行、支持增量重跑。subagent 写 `endpoint-{METHOD}-{slug}.md`；主 agent 仅撰写 aggregator 索引，不直接产出接口内容。
- **规模较大时分级聚合**：项目较大时在 aggregator 层增加一级子功能索引（`features-{slug}.md`）；接口子任务始终遵循"每接口一个 subagent"。
- **复杂流程辅以示意图**：多服务 / 多分支 / 异步 / 状态流转时，在描述末尾追加一张 mermaid 图。
- **完整捕获请求**：REST/RPC/MQ 等结构化输入必须交代请求维度——path 变量 / query / 关键 header（鉴权、Content-Type）/ body。body 为 JSON 时给出字段树（记号见步骤 4）；DTO 类型须递归追溯到定义读出真实字段，禁止臆测。
- **追踪入参流向**：跟踪每个请求参数（path/query/header/body 字段，含 DTO 嵌套字段）是否流入下列**敏感操作位置**——拼接到路径、拼接到命令、拼接到 URL、写入文件（非日志）、拼接到 SQL。**有则使用 `**输入流向**` 一行单独标注**，按"参数 → 操作位置（文件:行号）—— 类别"格式书写。**仅描述事实（"X 拼接到 Y"），不下风险结论**。**写日志（log.info/log.error 等）一律忽略**——使用过于普遍，信噪比低。
- **细化到具体动作与目标**：文件 / 命令 / 网络操作必须给出具体**动作**（读/写/追加/删除/拷贝/exec/POST/...）与具体**目标**（完整路径 / 命令行 / URL，含 query 与关键 header）；动态拼接部分以 `/x/{var}` 模板表达并标明变量来源。**禁止**"读配置"、"调网关"、"执行脚本"等抽象表述。

## 取舍

**保留**：用户输入入口、文件读写、外部进程、网络外呼、加解密/签名、鉴权与会话、数据持久化。
**剔除**：CSS/i18n/纯计算/单测/构建脚本/健康检查/静态资源/Spring Actuator 等纯运维端点。

## 方法

### 0. 加载工作目录

在 `<cwd>/biz-flow-recon/` 下查找：若存在 `knowledge/` 则**完整读取其中所有 .md 文件**作为先验；模板路径优先使用 `<cwd>/biz-flow-recon/templates/{name}.md`，回退至 skill 包内默认。`output/` 仅在产物落盘时再行创建。

### 1. 确定粒度

- **粒度 B（默认）**：子功能 + 接口。模板 `default.md`，使用完整骨架。
- **粒度 A**：用户明确提出"整项目/全量梳理/系统全貌"时。模板 `overview.md`。
- **粒度 C**：用户给出具体 URL 或 类#方法 时。模板仍为 `default.md`，但**仅截取一个接口块**（将 `#### POST /api/...` 提升至顶层 `# POST /api/... — 一句话功能`），不保留顶层叙述与子功能分组。

### 2. 枚举入口

识别 `@RestController`/`@Controller`/`@RequestMapping`、`@Scheduled`、MQ 监听器、CLI `main`、前端 router。逐条记录 `类#方法`、文件:行号，并按目录/包/模块初步划分子功能；当接口数超过 6 个子功能或 25 个接口时，先写出一份 `output/_plan.md`（格式见模板）供用户确认拆分方案。

### 3. 派发 subagent — 每接口一个独立任务（核心步骤）

对步骤 2 列出的**每一个接口**，使用 Task 工具启动独立 subagent，传入：
- skill 路径（用于读取 SKILL.md 与模板）
- 单接口 scope（`类#方法`、文件:行号、起始包范围）
- 输出文件路径 `<cwd>/biz-flow-recon/output/endpoint-{METHOD}-{path-slug}.md`
- knowledge / templates 路径

subagent 内部按下文步骤 4 完成单接口分析并落盘。**默认并行**派发；当 `conventions.md` 配置 `执行模式: 串行` 时改为串行。**已存在的 `endpoint-*.md` 默认予以跳过**——仅补做缺失项；如需重做请先删除对应文件。

主 agent 不直接撰写接口内容，待全部 subagent 完成后执行步骤 5。

### 4. （subagent 内部）追踪调用链、递归追溯、撰写描述

**本步骤为 subagent 的执行剧本**。每个 subagent 收到单接口 scope，独立执行：

追踪调用链至底层（Service/Mapper/配置/I/O/进程/网络/加密点）。遇到指向更深层代码的引用即予以追溯——常见目标：进程脚本、SQL 文件、内部 HTTP/RPC、消息 topic、反射类、动态加载的规则文件。

- **边界**：当前仓 / 本地兄弟目录 → 必须追溯；跨仓 → 仅当 `knowledge/conventions.md` 指明位置时追溯；真正外部（第三方/SaaS）→ 仅描述对外协议。
- **深度**：默认 2 层；第 3 层及以后以一句话概括。`conventions.md` 配置深度时按用户配置。
- **未能追踪到目标**：在描述中显式标注"未在工作区找到 X"，并加入文件末尾"未能追溯的引用"节。

按 `default.md` 单接口块格式撰写产物（顶层 `# {METHOD} {URL} — 一句话功能` 标题 + 描述/示意图/字段化列表 + 末尾未能追溯的引用节）。每条接口必须交代：

**具体性要求**（"细化到具体动作与目标"原则的落地）：
- **文件 I/O**：动作（读/写/追加/删除/拷贝/移动/列目录）+ 完整路径（动态部分以 `{var}` 标明变量来源）+ 格式（YAML/JSON/CSV/二进制/Logback 行日志等）
- **命令**：完整命令行（动态参数以 `{var}` 标明变量来源）+ 执行方式（`Runtime.exec`/`ProcessBuilder`）+ `文件:行号`
- **第三方**：`METHOD URL`（动态部分模板化）+ Content-Type + 关键 Header（鉴权、签名）+ body 概要 + 客户端 + `文件:行号`；复杂调用使用块状形式

呈现形式按流程特点选择：

- **mermaid 示意图**：多参与方 / 分支 / 异步——`sequenceDiagram` 用于多方调用链（最常用），`flowchart TD` 用于分支，`stateDiagram-v2` 用于状态机。每接口至多一张图，仅画主干 + 关键分支。
- **编号步骤**：内在顺序的操作序列。
- **字段化短列表**：多类独立事实，书写为 `**第三方**: ...` `**加解密**: ...`，**仅列出实际发生的类别**。
- **连贯叙述**：事实可自然串接成句时。
- **一句话**：简单读接口。

递归追溯所得的下游事实写入同一条描述。**不使用嵌套子标题**。

**请求参数记号**（Body 字段树通用约定）：

- `string!` / `int!` —— `!` 表示非空/必填（来源于 `@NotNull`/`@NotBlank` 等）
- `string?` —— 可选/可为空
- `[ ... ]` —— 数组
- `// ...` —— 行内注释，标注约束（长度、正则、范围）或字段含义
- 末尾附 DTO 全限定名 + `文件:行号`

简单 query 使用一行 inline（`**请求**: query \`?page=1&size=20\` ...`）；嵌套或字段较多时使用 fenced 代码块（` ```json5 ... ``` `）。GET 请求无 body 时不写 body 行。

### 5. 撰写 aggregator 索引（主 agent 执行）

全部 subagent 完成后，主 agent 撰写索引 aggregator（**不重复接口正文**，仅一句话概述 + 链接）：

| 粒度 / 规模 | aggregator | 内容 |
|---|---|---|
| C 单接口 | （无） | 产物即对应的 `endpoint-*.md` |
| B 限定（如"支付下单"） | `features-{slug}.md` | 子功能一句话概述 + 列出范围内的 `endpoint-*.md` 链接 |
| B 默认 / A 小型项目 | `features.md` 或 `overview.md` | 核心总结 + 按子功能分组列出全部 `endpoint-*.md` 链接 |
| A 大型项目 | `features-{slug}.md`（每子功能一份）+ `overview.md` | 三层索引：每个子功能一份索引列出其 `endpoint-*.md`；`overview.md` 列出各 `features-{slug}.md` |

aggregator 末尾合并各 `endpoint-*.md` 的"未能追溯的引用"节（去重后集中列出）。

**输出文件名约定**（slug 小写连字符，中文转拼音/英文同义词；**默认覆盖同名文件**便于 git diff）：

- `endpoint-{METHOD}-{path-slug}.md` — 每接口一份（**必产出**）
- `features.md` — 粒度 B 默认全量索引
- `features-{slug}.md` — 限定范围或大型项目子功能索引
- `overview.md` — 整项目顶层索引
- `_plan.md` — 子任务计划（步骤 2 产出）

完成后向用户告知产物路径（优先指出 aggregator 入口文件）。

## 工作目录与模板

`docs/workspace-layout.md` 详述被分析项目内 `biz-flow-recon/{knowledge,templates,output}/` 的约定。**每次输出前必须读取对应模板后再行撰写**，不得依赖记忆产出——以确保用户对模板的修改能即时生效。
