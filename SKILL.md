# biz-flow-recon

把代码讲给安全测试人员听——按子功能 + 接口的粒度，用最清晰的形式说"它在做什么"：文件 I/O、外部命令、第三方交互、加解密。**只描述事实**，判断权留给读者。

## 何时使用 / 何时不用

**使用**——只要用户表达出"想搞清这个代码在做什么"的意图，无论是否提到"安全"：
- "梳理业务流"、"讲讲这个项目 / 模块 / 接口"、"给我个 brief"、"帮我了解一下"
- "渗透测试前摸底"、"代码审计前盘点"、"接口/功能盘点"
- 具体到类或接口："讲讲 XxxController 在干什么"、"POST /api/yyy 这个接口怎么走的"
- 主要面向 **Java 后端**，前端与全栈也支持

**不使用**——这些走 `/security-review`：在做漏洞分析、写修复建议、写报告，或问"有没有 SQL 注入 / 越权 / XSS"等具体漏洞。

## 原则

- **描述事实，不评判**。禁用"风险/漏洞/不安全/缺乏校验/可能被攻击/建议加固"等词。
- **清晰优先，不拘形式**。按内容挑：图 / 编号 / labeled 短列表 / 句子 / 一行话。**禁止机械填表**（每条都 `- 入口:` `- 触发者:`），**禁止空字段**（"外部命令: 无"）。
- **优先跨信任边界的动作**（用户输入入口、I/O、外呼、加解密），而非内部纯计算。
- **能下钻就下钻**：调脚本读脚本、调内部接口找接收方、引用 SQL 读 SQL。
- **找不到必须报**：在描述里点名，并在产物末尾的 `## 未跟到的引用` 节集中列出。**不编、不漏、不静默吞掉**。
- **每接口一个 subagent**（强约束）：**每个接口的分析必须用独立的 Task subagent 完成**——互不影响、可并行、便于增量重跑。subagent 写 `endpoint-{METHOD}-{slug}.md`；主 agent 只写 aggregator 索引，不直接产出接口内容。
- **大就拆**：项目大时按子功能在 aggregator 层加一级索引（`features-{slug}.md`）；接口子任务的拆分一直就是按"每接口一个 subagent"做的。
- **复杂时配图**：多服务/多分支/异步/状态流转时，在描述后追加一个 mermaid 图。
- **捕请求**：REST/RPC/MQ 等结构化输入必须交代请求维度——path 变量 / query / 关键 header（鉴权、Content-Type）/ body。body 是 JSON 时给字段树（记号见步骤 4）；DTO 类型要下钻到定义读出真实字段，不要猜。
- **追入参流向**：跟踪每个请求参数（path/query/header/body 字段，含 DTO 嵌套字段）有没有落到这几类**敏感槽**——拼路径、拼命令、拼 URL、写文件（非日志）、拼 SQL。**有就用 `**输入流向**` 一行单拎出来强调**，按"参数 → 槽（文件:行号）—— 类别"写。**只描述事实（"X 拼到 Y"），不下风险结论**。**写日志（log.info/log.error 等）一律忽略**——太普遍、噪声大。
- **细到具体**：文件 / 命令 / 网络操作必须写出具体**动作**（读/写/追加/删除/拷贝/exec/POST/...）和具体**目标**（完整路径 / 命令行 / URL，含 query 与关键 header）；动态拼接的部分写成 `/x/{var}` 模板并标变量来源。**禁止**"读配置"、"调网关"、"执行脚本"这种抽象写法。

## 取舍

**保留**：用户输入入口、文件读写、外部进程、网络外呼、加解密/签名、鉴权与会话、数据持久化。
**砍掉**：CSS/i18n/纯计算/单测/构建脚本/健康检查/静态资源/Spring Actuator 等纯运维端点。

## 方法

### 0. 加载工作目录

在 `<cwd>/biz-flow-recon/` 找：若有 `knowledge/` 就**读完所有 .md** 当先验；模板路径优先走 `<cwd>/biz-flow-recon/templates/{name}.md`，回退到 skill 包内默认。`output/` 等到要写产物时再建。

### 1. 定粒度

- **粒度 B（默认）**：子功能 + 接口。模板 `default.md`，用整份骨架。
- **粒度 A**：用户明确说"整项目/全量梳理/系统全貌"时。模板 `overview.md`。
- **粒度 C**：用户给了具体 URL 或 类#方法 时。模板仍是 `default.md`，但**只取一个接口块**（把 `#### POST /api/...` 提到顶层 `# POST /api/... — 一句话功能`），不要顶层叙述和子功能分组。

### 2. 盘入口

找 `@RestController`/`@Controller`/`@RequestMapping`、`@Scheduled`、MQ 监听器、CLI `main`、前端 router。每条记 `类#方法`、文件:行号，同时按目录/包/模块初步分组成子功能；接口数 ≥ 6 个子功能或 25+ 接口时，先写一份 `output/_plan.md`（格式见模板）让用户先确认拆分。

### 3. 派 subagent — 每接口一个（核心步骤）

对步骤 2 列出的**每一个接口**，用 Task 工具起独立 subagent，传入：
- skill 路径（让它能读 SKILL.md 和模板）
- 单接口 scope（`类#方法`、文件:行号、起始包范围）
- 输出文件路径 `<cwd>/biz-flow-recon/output/endpoint-{METHOD}-{path-slug}.md`
- knowledge / templates 路径

subagent 内部按下面步骤 4 完成单接口分析并落盘。**默认并行**派发；`conventions.md` 写 `执行模式: 串行` 时改串行。**已存在的 `endpoint-*.md` 默认跳过**——只补做缺的；要重做先删对应文件。

主 agent 不直接写接口内容，等所有 subagent 完成后跑步骤 5。

### 4. （subagent 内部）跟链路、下钻、写描述

**这一步是给 subagent 看的剧本**。每个 subagent 收到一个接口的 scope，独立执行：

跟链路到底（Service/Mapper/配置/I/O/进程/网络/加密点）。遇到指向更深层代码的引用就跟——常见目标：进程脚本、SQL 文件、内部 HTTP/RPC、消息 topic、反射类、动态加载的规则文件。

- **边界**：当前仓 / 本地兄弟目录 → 必跟；跨仓 → 仅 `knowledge/conventions.md` 指明位置时跟；真正外部（第三方/SaaS）→ 描述对外协议即可。
- **深度**：默认 2 层；第 3 层及以后一句话概括。`conventions.md` 写明深度时按用户。
- **走不通**：在描述里点名"未在工作区找到 X"，并加入文件末尾"未跟到的引用"节。

照 `default.md` 单接口块格式写产物（顶层 `# {METHOD} {URL} — 一句话功能` 标题 + 描述/图/labeled 列表 + 末尾未跟到的引用节）。每条接口必须交代：

**具体性要求**（"细到具体"原则的落地）：
- **文件 I/O**：动作（读/写/追加/删除/拷贝/移动/列目录）+ 完整路径（动态部分 `{var}` 标变量来源）+ 格式（YAML/JSON/CSV/二进制/Logback 行日志等）
- **命令**：完整命令行（动态参数 `{var}` 标变量来源）+ 执行方式（`Runtime.exec`/`ProcessBuilder`）+ `文件:行号`
- **第三方**：`METHOD URL`（动态部分模板化）+ Content-Type + 关键 Header（鉴权、签名）+ body 概要 + 客户端 + `文件:行号`；复杂时用块形式

形式按流程特点挑：

- **mermaid 图**：多参与方 / 分支 / 异步——`sequenceDiagram` 用于多方调用链（最常用），`flowchart TD` 用于分支，`stateDiagram-v2` 用于状态机。一图最多一张，主干 + 关键分支。
- **编号步骤**：内在顺序操作。
- **labeled 短列表**：多类独立事实，写成 `**第三方**: ...` `**加解密**: ...`，**只列发生过的**。
- **几句话**：自然能串成句子时。
- **一行话**：简单读接口。

下钻得到的下游事实写在同一条描述里。**不要嵌套子标题**。

**请求参数记号**（Body 给字段树时用，紧凑统一）：

- `string!` / `int!` —— `!` 表示非空/必填（来自 `@NotNull`/`@NotBlank` 等）
- `string?` —— 可选/可为空
- `[ ... ]` —— 数组
- `// ...` —— 行内注释，写约束（长度、正则、范围）或字段含义
- 末尾挂 DTO 全限定名 + `文件:行号`

简单 query 1 行 inline（`**请求**: query \`?page=1&size=20\` ...`）；嵌套或字段多用 fenced 代码块（` ```json5 ... ``` `）。GET 没 body 就不写 body 行。

### 5. 写 aggregator — 主 agent 做

所有 subagent 完成后，主 agent 写索引 aggregator（**不重复接口正文**，只一句话概述 + 链接）：

| 粒度 / 规模 | aggregator | 内容 |
|---|---|---|
| C 单接口 | （无） | 产物即那一份 `endpoint-*.md` |
| B 限定（如"支付下单"） | `features-{slug}.md` | 子功能一句话概述 + 列出范围内的 `endpoint-*.md` 链接 |
| B 默认 / A 小项目 | `features.md` 或 `overview.md` | 核心总结 + 按子功能分组列出所有 `endpoint-*.md` 链接 |
| A 大项目 | `features-{slug}.md`（每子功能一份）+ `overview.md` | 三层索引：每个子功能一份索引列其 `endpoint-*.md`；`overview.md` 列各 `features-{slug}.md` |

aggregator 末尾合并各 `endpoint-*.md` 的"未跟到的引用"节（去重后集中列出）。

**输出文件名约定**（slug 小写连字符，中文转拼音/英文同义词；**默认覆盖同名**便于 git diff）：

- `endpoint-{METHOD}-{path-slug}.md` — 每接口一份（**必产出**）
- `features.md` — 粒度 B 默认全量索引
- `features-{slug}.md` — 限定范围或大项目子功能索引
- `overview.md` — 整项目顶层索引
- `_plan.md` — 子任务计划（步骤 2 写）

写完告诉用户产物路径（先点出 aggregator 入口文件）。

## 工作目录与模板

`docs/workspace-layout.md` 详述被分析项目里 `biz-flow-recon/{knowledge,templates,output}/` 的约定。**每次输出前先读对应模板再照样写**，不要凭记忆——这样用户改了模板能立即生效。
