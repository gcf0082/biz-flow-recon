# biz-flow-recon

像一位熟悉系统的工程师，把代码讲给安全测试人员听：先用一段连贯的话讲清"它在做什么"，再按子功能分组列出**实际存在的事实**——文件 I/O、外部命令、第三方交互、加解密。判断权留给读者。

## 原则

- **描述事实，不评判**。禁用"风险/漏洞/不安全/缺乏校验/可能被攻击/建议加固"等词。
- **不存在的事实就不写**。这条业务流没有外部命令？就不出现"外部命令"那一行。让产物像讲解，不像填表。
- **优先跨信任边界的动作**（用户输入入口、I/O、外呼、加解密），而不是内部纯计算。

## 取舍

**保留**：用户输入入口、文件读写、外部进程调用、网络外呼、加解密/签名、鉴权与会话、数据持久化。

**砍掉**：纯 CSS/样式、i18n 文案、内部纯计算函数、单元测试、构建脚本、健康检查、静态资源、Spring Actuator 之类纯运维端点。

## 方法

1. **定粒度**（默认 B）：
   - **默认走粒度 B**：按子功能分组（登录、下单、文件上传……），每组下列出相关接口。日常讲解最自然的层级。
   - **粒度 A（整项目）**：仅当用户说"整个项目""全量梳理""系统全貌""所有功能"等明确要总览时使用。
   - **粒度 C（单接口）**：仅当用户给了具体 URL（`POST /api/xxx`）或类#方法名时使用。
2. **盘入口**：在选定范围内找 `@RestController`/`@Controller`/`@RequestMapping`、`@Scheduled`、MQ 监听器、CLI `main`、前端 router。
3. **跟链路**：每个入口跟到底——Service、Mapper/Repository、配置文件（`application.yml`）、文件 I/O、外部进程、网络调用、加密点。
4. **抓事实**：只记录**实际发生**的动作，附代码位置（`类#方法`，`文件:行号`）。
5. **输出**：按下面模板写。

## 字段约定

每条业务流（接口/方法）的字段：

- **必填**：入口、触发者、代码位置
- **按需**（有就写、没有就略）：文件读写、外部命令、第三方交互、加解密

不要为了凑模板写"外部命令：无""加解密：无"。读起来要像讲解。

## 输出模板

三种粒度共同遵守一个原则：**开头一段连贯的自然语言讲解，再按子功能分组列事实**。

### 粒度 B — 默认（子功能 + 接口）

```
# {项目名 / 你正在讲的范围} 业务流讲解

## 整体在做什么
80-200 字连贯叙述：这个范围内的代码完成什么业务、由谁触发、关键流程怎么串。
不要 bullet，成段叙述。

## 业务流

### 子功能 1：用户登录与会话

#### POST /api/login
- 入口：com.acme.auth.AuthController#login（AuthController.java:42）
- 触发者：未登录用户
- 第三方交互：调 https://idp.example.com/oauth/token，POST application/x-www-form-urlencoded，
  Header Authorization: Basic <base64(client_id:secret)>，body grant_type=password&...。
  客户端 RestTemplate（OAuthClient.java:31）
- 加解密：BCrypt.checkpw 校验密码哈希——避免明文比对；JJWT 以 HS256 签发 Token，
  密钥取自 JWT_SECRET 环境变量——颁发会话凭证
- 文件读写：读 config/oauth.yaml（YAML，OAuth 客户端配置）；写
  /var/log/acme/auth.log（Logback 行日志）
- 代码位置：src/main/java/com/acme/auth/

#### GET /api/users/me
- 入口：com.acme.user.UserController#me（UserController.java:18）
- 触发者：已登录用户
- 代码位置：src/main/java/com/acme/user/
```

注意 `/api/users/me` 这个接口下**只有 3 行**——它就是个查当前用户的简单读接口，没有文件 I/O / 命令 / 第三方 / 加解密。**就让它简短**，不要硬凑字段。
